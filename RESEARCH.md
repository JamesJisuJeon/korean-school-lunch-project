# 🏫 동남학교 간식 관리 시스템(Korean School Lunch Project) 상세 연구 보고서

본 프로젝트는 동남학교의 점심/간식 주문 및 수납을 효과적으로 관리하기 위해 개발된 시스템입니다. 기존의 엑셀 기반 수기 관리를 탈피하여 모바일과 데스크톱을 모두 지원하는 현대적인 웹 애플리케이션으로 구축되었습니다.

---

## 1. ⚙️ 기술 스택 (Tech Stack)

*   **프레임워크:** Next.js 16.2.2 (App Router 기반)
*   **프로그래밍 언어:** TypeScript
*   **데이터베이스 & ORM:** PostgreSQL, Prisma (`@prisma/client` 6.x)
*   **인증 연동:** NextAuth.js (v5 beta) 기반 자가 인증 처리 로직
*   **스타일링:** Tailwind CSS v4, `next-themes`를 활용한 라이트/다크모드 지원
*   **데이터 시각화/UI 요소:** `lucide-react` (아이콘), 커스텀 모달 및 카드 뷰
*   **인프라 및 기타:** PWA(Progressive Web App) 설치 지원, WebP 이미지 리사이징(`sharp`), `clsx`, `tailwind-merge`

## 2. 👥 핵심 권한 및 역할 (Role-based Access Control)

시스템은 `User` 모델을 중심으로 크게 4가지 권한을 부여하며 기능 접근을 엄격히 제한합니다:

1.  **ADMIN (시스템 관리자):**
    *   사용자 계정 엑셀 대량 등록 및 권한 부여
    *   학년도(Academic Year) 및 학급(Class) 마스터 데이터 관리
    *   전체 학생 및 배정 관리
    *   (실수 방지용 '시스템 관리 모드' 토글 기능 지원)
2.  **PA (학부모회 - Parents Association):**
    *   **메뉴 등록 및 게시:** 주간 간식(Main, Dessert, Beverage) 메뉴 생성, 이미지 업로드, 게시 여부 제어
    *   **현장 수납 및 매점 관리:** 당일 현장 신청, 결제(PAID/UNPAID/WAITING) 상태 실시간 추적, 학생별 $5 매점 쿠폰 처리
    *   **운영 집계 및 통계:** 반별 통계 및 매출, 수납 대기/취소, 무료 간식 및 PA 자녀 데이터 실시간 추적 대시보드 열람
3.  **TEACHER (담임/보결/보조 교사):**
    *   배정받은 학급의 학생 명단 및 간식 신청 현황 조회
    *   학생 건별 '배식완료(isServed)' 상태 체크를 통해 실제 배급 여부 추적
4.  **PARENT (학부모):**
    *   본인 계정에 연동된 자녀 리스트 관리 (다자녀 지원)
    *   게시된 해당 주차의 간식 메뉴 주문 (NZ 시간 기준 목요일 낮 12시 기한 자동 마감)
    *   주문 취소 및 특이사항(알러지 등) 메모 기록

## 3. 💾 데이터베이스 스키마 구조 분석

Prisma 스키마는 시스템의 모든 엔티티를 유기적으로 연결하고 있습니다:

*   **User & Student:** 1명의 User(학부모)는 여러 Student를 다대다 관계로 가질 수 있어 다자녀 관리가 가능합니다. 학부모회 소속 자녀는 `isPAChild` 플래그로 구별되어 무료 간식 등의 혜택을 받습니다.
*   **Class & AcademicYear:** 학급은 반드시 특정 학년도에 종속됩니다. `sortOrder`를 이용해 학급 목록의 노출 순서를 정교하게 제어합니다. 한 명의 `User`는 담임(`teacher`)일 수도, 보결(`Substitute`)일 수도, 그리고 보조교사(`ClassAssistant`)일 수도 있습니다.
*   **Menu:** 간식의 실제 엔티티입니다. 토요일을 기준으로 날짜가 기록되며 이미지(poster), 메인/디저트/음료 내역, 공지사항을 포함합니다. `isPublished`가 true여야만 보입니다.
*   **Order:** 학생(`Student`)과 메뉴(`Menu`) 간의 간식 주문 정보입니다. `PRE_ORDER`(사전 신청)와 `ON_SITE`(현장 신청)로 나뉘며, 납부 상태(`status`), 배식 여부(`isServed`), 그리고 특이 사항 메모가 저장됩니다.
*   **CouponSale:** 당일 현장 구매용 매점 쿠폰 구매 이력($5 고정가 기준)을 추적합니다.

## 4. 🚀 핵심 비즈니스 로직 및 워크플로우

### 4.1 간식 주문 및 마감 자동화
메뉴의 날짜가 특정 '토요일'로 정해지면, 시스템 내 시간 로직은 자동으로 해당 주차 '목요일 낮 12시(뉴질랜드 시간)'를 마감 시간으로 설정합니다. 마감 전까지 학부모는 "수납대기(WAITING)" 상태 내에서 자유롭게 신청/취소가 가능하지만, 마감 이후엔 변경이 통제됩니다.

### 4.2 수납금액 정산 플로우
학생이 PA(학부모회) 자녀인지 여부, 혹은 메뉴가 기본적으로 책정한 가격에 따라 결제금이 동적으로 달라집니다:
*   수납상태: `WAITING` (대기) → `PAID` / `POST_PAID` (수납 완료) → `UNPAID` (후납) → `CANCELLED` (취소).
*   이러한 상태는 PA 대시보드(집계 화면)에서 총 매출 합계, 반별 납부/후납 금액, 사전/현장 주문 건수를 집계할 때 사용됩니다.

### 4.3 모바일 현장 운영 최적화
토요일 현장에서 폰이나 태블릿으로 시스템을 활용하는 시나리오를 철저히 고려한 UI가 적용되어 있습니다. 
*   **PWA 적용:** 모바일의 홈 화면에 설치(Add to Home Screen)할 수 있어 네이티브 앱처럼 동작합니다.
*   **반응형 UI (Card UI):** 넓은 표(Table) 대신 세로 모바일 화면에 맞는 카드 리스트(모바일 현장 수납), 3열 통계 대시보드 등을 분리하여 구현했습니다.
*   **시각적 식별 도구:** PA(학부모회) 자녀는 고유 색상의 배지나 노란 배경 처리로 현장 요원이 직관적으로 인지할 수 있게 돕고 있습니다.

## 5. 💡 향후 유지보수 및 주요 포인트

1. **시간대(Timezone) 유의:** 해외(뉴질랜드) 특성을 반영한 마감 시간 분기 로직이 구현되어 있으므로, 서버 환경 변수에 의존하기보다는 로케일(Locale) 코드에 주의해야 합니다.
2. **다크 모드 일관성:** 모바일/데스크톱은 물론 로그인 화면, 컴포지트 컴포넌트 전체에 `.dark` 클래스 체인으로 대응되었습니다. iOS Safari의 날짜 피커 등 브라우저 네이티브 컴포넌트까지 대응되어 있으므로 CSS 수정 시 주의가 필요합니다.
3. **오프라인 및 안정성:** Service Worker가 내장되어 PWA를 돕지만 데이터 패칭은 필수이므로 안정적 네트워크 확보가 요구됩니다.
4. **정기 이관 작업:** 매 학기 초(뉴질랜드 학사 일정), 학년도 및 학급 정보를 일괄 업데이트해야 하는 주요 관리 업무 플로우가 필요합니다.

## 6. 🚨 코드 품질 및 스타일/리팩토링 권고사항

현재 코드베이스는 안정적으로 동작하지만, 향후 유지보수성 및 확장성을 높이기 위해 아래와 같은 로직 중복 및 UI 디자인 불일치 요소들을 리팩토링할 것을 강력히 권장합니다.

### 6.1 컴포넌트 마크업(Markup) 중복 — 반응형 디자인 안티패턴
* **이슈 (SalesManagementClient 등):** 반응형 웹을 구현할 때 `xl:hidden`(모바일/태블릿용 카드 뷰) 영역과 `hidden xl:block`(데스크톱용 테이블 뷰) 영역에 **동일한 상태 변경 로직과 이벤트 핸들러(onClick, onChange 등) 마크업이 거의 그대로 복사/붙여넣기** 되어 있습니다. 예를 들어, 쿠폰의 `+`, `-` 버튼 로직 및 렌더링 코드가 두 구역에 중복 존재합니다.
* **권고사항:** CSS Grid/Flexbox의 조건부 레이아웃이나 `hidden` 유틸리티에 의존하기보다, 중복되는 로직(예: 쿠폰 수량 제어 버튼 블록, 상태 표시 드롭다운)을 별도의 공통 하위 컴포넌트(예: `<CouponControl studentId={...} qty={...} />`)로 분리하여 한 곳에서만 관리해야 합니다. 

### 6.2 하드코딩된 상수(Constants) 파편화
* **이슈:** `PAYMENT_STATUSES` (예: `WAITING`, `PAID`, `UNPAID`, `POST_PAID`, `FREE_SNACK`) 상수와 관련된 색상(`bg-*`) 설정이 여러 페이지(`SalesManagementClient.tsx`, `TeacherClassClient.tsx`, `ClassOrdersClient.tsx`, `AnalyticsClient.tsx`)에 걸쳐 로컬 변수로 하드코딩 및 재정의되어 있습니다. 
* **권고사항:** `src/lib/constants.ts` 에 상태값 배열과 통합 UI 색상 맵을 정의(`export const PAYMENT_STATUS_MAP`)하고, 이를 필요한 페이지에서 공통적으로 import하여 사용하도록 즉시 통합해야 합니다.

### 6.3 필터 및 복구(초기화) 로직의 반복
* **이슈:** `classList` 추출(반 필터용 드롭다운 목록 추출 로직) 및 다단계 필터(`filterClass`, `filterStatus`, `filterCoupon` 등) 로직, 새로고침/정렬 버튼 디자인 패턴 코드가 각 관리 페이지마다 동일한 형태로 존재합니다.
* **권고사항:** 필터링 바(Filters Bar)와 테이블 헤더의 정렬 버튼, 새로고침(`RefreshCw`) 버튼은 애플리케이션 공용 컴포넌트(`<Toolbar />` 또는 `<FilterSelect />`)로 추상화하여 중복 코드를 극적으로 줄여야 합니다.

### 6.4 API 엔드포인트 내 상태 불일치 처리
* **이슈:** 클라이언트에서 UI 상의 악의적이거나 예기치 않은 데이터 입력 방지를 위해, 서버(API) 측에서 `Amount`(가격)이나 상태(`Status`)가 유효한지에 대한 철저한 유효성 검사(Zod 등을 통한 검증)가 다소 산재해 있을 수 있습니다. (예: FREE_SNACK 상태인데 납부 금액이 $0이 아니거나, PA Child가 아님에도 혜택 가격이 들어오는 경우 차단 등)
* **권고사항:** 비즈니스 규칙은 철저히 백엔드 API 단(혹은 공용 유틸 파트)에 캡슐화하여 프론트엔드가 단순히 이를 호출하게끔 책임(Responsibility)을 분리해야 데이터 무결성을 확실히 챙길 수 있습니다.
